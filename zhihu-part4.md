# 第 4 篇：为什么我会说 NetPilot 更像平台骨架，而不是功能堆叠？

写到第 4 篇，基本就是收口了。

前面几篇分别讲了定位、讲了监控和自动化，也讲了配置、合规、拓扑。现在再回头看，更该问的其实已经不是“它有没有这些功能”，而是另一件事：它到底像不像一套平台。

我自己判断一个项目是不是平台，不太看功能列表有多长。

我更看它后面的更新是在干什么。

如果只是不断往上叠页面、叠按钮、叠模块，那大概率只是功能越来越多。

但如果它开始补统一对象、统一流程、统一交互，开始在乎状态连续不连续、恢复闭环不闭环、重启之后会不会失忆，那味道就不太一样了。

按 NetPilot 现在这个节奏，我会更倾向于后者。

## 一个人能用的工具，和一群人能用的平台，不是一回事

很多脚本、小工具，个人用起来其实都挺爽。

问题是一进团队环境，马上就会冒出一堆现实问题：

- 谁能做什么。
- 谁改了什么。
- 出了事以后，责任边界怎么讲。
- 新人接过来以后，看不看得懂之前的动作。

这些东西如果没有权限和审计去兜，平台就很难真正落地。

NetPilot 这块做得不算重，但边界已经有了。现在基本能看到的是三类角色：

- Administrator
- Operator
- Viewer

我反而觉得这种克制是对的。内部平台最怕的不是权限做得不够复杂，而是为了“专业感”把自己设计死。

而且用户管理也不是孤零零挂着的。它跟审计日志其实是一组东西：前者管谁能进、谁能改，后者管谁改了、改完结果怎样。两页放在一起，团队协作这件事才算有了基本盘。

## 审计日志这东西，平时觉得普通，团队一大就离不开

很多人对日志的理解还停在“出事以后查一下”。

其实审计日志更像团队协作的底账。

它解决的事情至少包括：

- 出问题时能不能回溯。
- 做过变更之后能不能留痕。
- 多人协作时能不能讲清楚责任边界。
- 后面复盘流程时能不能看到真实历史。

说到底，一个团队越成熟，就越不能把关键动作只留在人的记忆里。

NetPilot 在这块的意义，不是多了张日志表，而是让这些动作开始变成系统对象，而不是“做完就过去”的临时信息。

如果把前面 3 篇连起来看，4 篇里其实已经把主导航那几块都走过了：Dashboard 负责开工前的全局一眼，Monitoring 和 Health 负责发现问题，Alert Center 负责值班闭环，Inventory 负责对象底座和接口横切面，Automation 负责动作、场景和历史，Config Center 和 Configuration 负责配置基线与发布准备，Compliance 负责标准，Topology 负责关系，Users 和 Audit Logs 负责团队边界。再加上顶部通知中心和用户菜单这些公共入口，日常使用会碰到的主区域基本都带到了。

## 技术选型这块，我会说它挺务实

如果你本身就在做内部平台，这项目还有个地方挺值得看，就是它的技术路线没有走花活。

前端是 React + TypeScript + Vite。

这套对中后台来说很常规，功能承载力够，迭代速度也够。图表和拓扑再配上对应库，基本能把前台该承的东西承下来。

后端是 FastAPI，默认存储是 SQLite。

很多人一看到 SQLite 就会下意识皱眉，觉得是不是太轻了。

可你真放到内部平台、实验室、PoC、中小规模落地这种场景里看，它反而挺合理：

- 部署门槛低
- 启动快
- 维护成本低
- 先把东西跑起来很容易

更重要的是，它虽然轻，但不是轻到只剩演示效果。现在已经开始把一些真正影响稳定性的上下文持久化下来，比如告警事件、协议邻居上一轮状态、宿主机资源趋势这些。说明它不是只想把页面跑通，而是在轻量部署的前提下，慢慢补运行态记忆。

网络自动化层面用的也都是比较稳的东西，Netmiko、Scrapli、SNMP、TextFSM。它们不一定新，但靠谱。

我一直觉得内部平台最怕的不是技术不新，而是技术太花，最后没人接得住。NetPilot 这条路的参考价值也在这儿：不用一上来把架构做得特别重，也能先把骨架搭出来。

## 它适合哪些团队，其实也挺明确

不是所有团队都需要这么一套东西。

但下面几类场景，我觉得会比较对路：

### 1. 企业内部网络运维团队

设备多、厂商杂、变更频繁，靠人肉串流程已经越来越吃力。这时候统一控制台的价值会很明显。

### 2. 实验室和测试环境

经常要做多厂商验证、批量执行、配置比对，这类平台会比纯脚本模式更容易管理。

### 3. 网络平台建设 PoC

如果你的团队正准备做自己的内部网络平台，这项目很适合拿来当骨架样本看。

### 4. 中小规模自动化落地

不想永远靠脚本堆叠，但也没必要一上来就上很重的商业平台，这个定位就比较合适。

## 所以我为什么一直说它更像平台，而不是工具集合

前面几篇其实一直在绕着这句话打转。

工具集合的特点是什么？

功能也许不少。
但彼此之间不成线。
很多上下文还是要靠人自己补。

平台的特点呢？

对象开始统一。
动作开始串起来。
历史能追。
标准能落。
团队能协作。

最近这轮告警中心调整就是个很典型的例子。表面看像前端收敛，实际上和它一起补上的，还有协议邻居告警的持久化与联动、BFD 解析回退、宿主机资源告警通知与恢复通知这些后台能力。

这些改动合起来指向的是一件很具体的事：平台开始在乎状态是不是连续的，恢复是不是完整的，进程重启以后会不会像失忆一样把上下文丢掉。

这时候你再看它，就不太像“又多做了一个模块”，而更像“底座开始慢慢成型”。

NetPilot 现在当然还不是那种很重的大平台。

但它已经不只是把几块功能拼起来了，而是在认真回答一个老问题：网络运维能不能少靠一点人肉硬串，多靠一点平台去接流程。

真要给整个系列收个尾，其实就一句话：

NetPilot 值得研究的，不是它做了多少功能，而是它已经开始把网络运维从“人盯工具”往“平台承接流程”这边推了。

这其实就是平台化最有价值的地方。

不是让工程师没用了，而是把工程师脑子里那些零散经验，慢慢沉淀成团队能反复使用、持续迭代的系统能力。

如果一个项目开始走到这一步，它就已经不只是“能用”，而是值得继续做下去。

## 最后留两个问题

如果你也是做网络运维、自动化平台或者内部工具建设的，其实可以回头想两件很现实的事：

- 你们团队现在的设备、配置、监控、审计，真的在一条工作流里吗？
- 如果没有统一控制台，工程师每天到底有多少时间耗在切系统和补上下文上？

如果这两个问题你越想越觉得别扭，那 NetPilot 这种项目为什么值得被单独拿出来讲，答案其实已经很接近了。
