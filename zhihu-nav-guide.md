# NetPilot 导航全功能导览：左侧这一栏，到底把哪些活收进来了？

前面那 4 篇，主要还是按能力主线在讲。

这样写比较容易讲清楚“为什么值得看”。

但也有个问题：如果你真想按导航去理解这个平台，看完以后脑子里未必有完整地图。

所以这篇补的，就是这张地图。

不谈什么大词，就按现在左侧导航和几个公共入口，挨个过一遍：它们到底都在干什么。

## 先回答一个直接问题：前面几篇有没有把导航讲全？

没有。

前面几篇确实已经讲到了这些主线：

- 仪表盘
- 监控中心
- 告警中心
- 自动化
- 配置管理
- 合规审计
- 用户和审计

但如果按真实导航结构来算，其实还有几块之前没有单独展开：

- 健康检测
- 拓扑页完整交互
- Inventory 里的接口监控子页
- Automation 里的场景库和执行历史
- Config Center 里的 Diff、搜索、定时备份
- Configuration 这个独立工作台

所以这篇就是把它们补齐。

## 先看结构：这条导航不是菜单堆，是工作分区

按现在界面看，左边这栏大概可以分成几类入口。

第一类，是点进去就是一级页的：

- 仪表盘 Dashboard
- 监控中心 Monitoring
- Topology
- 健康检测 Health Detection
- 合规 Compliance
- 系统配置 Configuration
- 审计日志 Audit Logs
- 用户管理 User Management

第二类，是折叠分组：

- 告警中心
- Inventory
- Automation
- Config Center

第三类，是每个分组里面的二级页。

再往上，其实还有两个经常被忽略、但实际很好用的公共入口：

- 顶部通知中心
- 用户菜单快捷入口

我自己会把这套东西理解成工作分区，而不是功能目录。它大概是在把几类典型工作拆开：看状态、找对象、做动作、查历史、管规则。

## 仪表盘：先看一眼全局，再决定往哪钻

仪表盘不是拿来深挖细节的。

它更像开工前先扫一眼全局。

这里承接的基本都是摘要：

- 设备规模和在线情况
- 任务执行和计划任务概览
- 合规趋势
- 平台宿主机资源状态
- 告警和运维状态概况

如果一个页面能先告诉你“今天大盘是不是稳的”，它就算合格。

> 截图位 2：仪表盘总览卡片区

## 监控中心：真正的实时工作台

监控中心是现在最像前线页面的地方。

它不是简单列设备，而是围绕“先搜设备，再看实时，再看趋势，再看当前告警”这条路来组织的。

这里核心能做的事情，大概是：

- 搜设备
- 看实时接口和趋势数据
- 联动当前监控告警
- 看宿主机资源状态
- 在实时和趋势视图之间切换

这页最像值班视角。先把对象找出来，再判断现在到底值不值得动。

> 截图位 3：监控中心搜索与趋势页

## 告警中心：现在终于像个独立模块了

这块现在已经不是“监控下边挂个告警页”那么简单了。

它被拆成了三页：

- 告警信息
- 告警规则
- 维护期

这三页其实刚好对应三种完全不同的工作。

### 告警信息

这是值班页。

适合用来：

- 看当前告警和状态
- 按级别、阶段、条件筛选
- 确认、跟进、补备注
- 从具体告警跳到维护期处理

### 告警规则

这是边界页。

它在定义什么情况该报，什么级别，阈值怎么设。

现在规则范围已经不只是接口 down、CPU、内存这种老几样了，还包括：

- 温度、风扇、电源
- 接口错误率、接口 flap
- BGP 邻居、OSPF 邻居、BFD 会话

### 维护期

这是降噪页。

变更、割接、巡检、检修这种场景，没有维护期基本很难活。

这页主要就是拿来：

- 查看维护窗
- 新建静默
- 复制已有维护窗
- 删除不需要的计划

这三页合在一起，告警中心才算真正成立。

> 截图位 4：告警信息 / 规则 / 维护期三页联图

## Topology：这页的重点不是图，是关系

Topology 页看起来很像“展示页”，但真要说它值钱的地方，还是关系视角。

它现在承接的大概是这些事：

- 基于 LLDP 的邻居发现
- 按站点、角色、状态筛拓扑
- 看链路健康、退化、中断、陈旧状态
- 看当前节点的邻居、链路和风险摘要
- 导出拓扑图

这页很适合排障，也适合变更前确认影响面。

只看设备表，你很难快速知道谁连着谁。拓扑图的价值，就是把这件事做得符合人脑理解方式。

> 截图位 5：拓扑画布与右侧详情栏

## 健康检测：不是监控页的重复版

这个入口很多人第一眼会觉得和监控中心很像。

但它俩焦点不一样。

监控中心偏实时数据和趋势。
健康检测偏设备健康状态摘要和问题筛查。

更适合回答的问题是：

- 哪些设备已经明显不健康了
- 哪些设备应该优先拉进监控页细看
- 哪些设备正在进入需要关注的状态

如果监控中心像放大镜，那健康检测更像体检筛查页。

## Inventory：别把它只当设备名册

Inventory 下面现在有两个子页：

- 设备列表
- 接口监控

### 设备列表

这个页当然是台账入口，但不是静态表。

这里能做的事情包括：

- 搜索和筛选设备
- 看设备状态、平台、站点、角色
- 导入导出
- 新增、编辑、删除
- 测试连接
- 打开设备详情

后面很多动作都得先从这里找对象。

### 接口监控

这个子页很容易被忽略，其实很好用。

它给的是跨设备视角，不是先点某台设备再看接口。

这里可以：

- 横向看一批设备的接口 UP / DOWN 摘要
- 看错误、丢包、带宽利用率
- 看 flap 和持续时间
- 做 SNMP 测试和手动同步
- 展开某台设备看接口明细表

如果监控中心更适合盯单台设备，这页更适合扫全局接口状态。

## Automation：动作、场景、历史，三件事被放到一起了

Automation 下面有三个子页：

- 直接执行
- 场景库
- 执行历史

### 直接执行

这是自动化主工作台。

不是一个裸命令框，而是会把目标设备、场景选择、变量配置、执行预览、风险确认这些步骤串起来。

适合做：

- 单设备或批量执行
- 套用内置场景
- 做 Dry Run / Validation
- 看前检查、后检查和回滚路径

### 场景库

这页更像知识库。

它把工程师常用动作收成一组可复用场景，你能看到：

- 内置场景
- 自定义场景
- 风险级别
- 支持平台
- 各阶段命令数量

### 执行历史

这页负责回答：刚才到底做了什么，结果怎么样。

可以：

- 搜历史
- 按状态筛
- 看正在运行或已完成的任务
- 看设备级结果和阶段输出

这页非常重要。自动化如果没有历史回放，很难让团队放心用。

## Config Center：历史配置这条线，被拆清楚了

Config Center 下面有四页：

- 备份历史
- 配置对比
- 配置搜索
- 定时备份

### 备份历史

看历史快照、手动拍快照、对全在线设备执行备份，这些都在这里。

### 配置对比

这是拿来看“到底改了哪几行”的。

可以搜快照、选两个版本做 Diff，还能按 interface、route、acl、bgp、ospf 这些关键词快速跳差异块。

### 配置搜索

这是全局检索入口。

搜 IP、网段、VLAN、路由、BGP、OSPF、ACL、NTP、SNMP 这类东西，会很顺手。

### 定时备份

这页更偏策略，重点在于周期备份是不是开着、概况怎么样、存储路径是不是规范。

这四页合起来，配置管理才不只是“我能下载个配置文件”。

## Compliance：这页开始有治理平台的感觉了

合规页不是拿来看个分数爽一下的。

它更像把“不规范”变成持续对象：

- 有得分
- 有设备审计总量
- 有开放问题数
- 有高危和严重问题数
- 有发现项列表
- 有筛选
- 有详情和跟进动作

也就是说，这页已经不只是在告诉你“有问题”，而是开始接整改链路了。

> 截图位 16：合规总览与发现项列表

## Configuration：这页不是重复，是另一层工作台

很多人容易把 Configuration 和 Config Center 搞混。

其实不是一回事。

Config Center 偏历史配置和快照。
Configuration 更像模板发布工作台。

它把这些东西揉到了一起：

- 模板资产
- 全局变量
- 模板源码编辑
- 渲染预览
- 发布前检查
- 发布范围筛选
- 目标设备预览
- 从模板转场景草稿
- 把模板提交到 Automation

一个偏历史和结果。
一个偏模板和发布准备。

> 截图位 17：Configuration 模板工作台

## 审计日志：这页负责把动作留下来

审计日志页不是日志堆，它更像可回看的操作历史。

现在能按这些维度筛：

- 分类
- 严重级别
- 状态
- 时间范围

还能看：

- 动作摘要
- 目标对象
- 操作人
- 状态和级别
- 详情

覆盖范围也挺广，认证、资产、执行、配置、Playbook、模板、合规这些都会进来。

这页的意义很朴素：很多动作不再只是做完就算了，而是能被系统记住。

## 十四、用户管理：平台开始从“自己能用”走向“团队能用”

## 十五、别忽略顶部通知和用户菜单，这也是一部分“导航”

严格说，顶部区域不算左侧导航的一部分，但从使用体验来说，它确实承担了全局导航作用。

这里主要有两类公共入口：

- 通知中心
- 用户菜单快捷入口

通知中心更像值班视角下的全局收件箱。你不一定停在告警中心页，但依然能通过顶部铃铛知道有没有新的告警和未读项。

用户菜单则更像一个快捷跳转区，它会把常用的页面入口重新挂出来，让用户不用每次都从左侧层层展开去找。

这两个位置看起来不像主功能，但它们对“平台是不是顺手”影响非常大。因为很多时候，用户并不是从头按导航走流程，而是从一个全局提醒直接跳进处理页。

## 十六、如果按工作流看，这套导航其实在组织 4 类工作

## 十七、如果你要发知乎，这篇最适合怎么放

这篇导览稿有两种最自然的放法。

第一种，是放在 4 篇系列前面，当导读。

这样做的好处是，读者先建立整体地图，后面再看每一篇展开，就不容易迷路。

第二种，是放在 4 篇系列后面，当补充篇。

这样做的好处是，前面 4 篇先讲价值和能力主线，最后再把导航和模块一次性讲全，比较像一个收口总结。

如果按知乎阅读体验来说，我个人会更偏向第一种。因为这个项目模块比较多，先给读者一张“地图”，理解成本会明显低很多。
