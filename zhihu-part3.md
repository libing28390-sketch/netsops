# 第 3 篇：为什么说配置、审计、拓扑这三块，才决定一个平台能不能进正式环境？

前两篇其实都在讲前线视角。

能不能看见问题。
能不能把动作做起来。

但真往正式环境里走，大家最后看的往往不是这些，而是另外一层东西：你做完动作以后，后面兜不兜得住。

最典型的就是三块：

- 配置管理
- 合规审计
- 拓扑发现

这三块不一定最显眼，但分量很重。很多系统前台看着挺热闹，一到这几块就开始发虚。

## 自动化做得越猛，越需要配置这层兜底

这个道理其实不复杂。

你当然可以改设备。
问题是，改之前是什么样，你知道吗？
改完之后到底动了哪几行，你看得快吗？
真翻车了，你退得回去吗？

这些问题如果说不清，自动化越强，风险只会越大。

所以配置管理在我这儿从来不是附加功能，更像安全带。

NetPilot 现在这块已经不是一个孤零零的“备份页”了，而是两层东西一起在跑。

一层是 Config Center，偏历史和结果，4 个入口都各管一摊：

- 配置备份
- 快照浏览
- Diff 对比
- 配置搜索
- 定时备份

另一层是 Configuration 工作台，偏模板和发布准备：

- 模板资产
- 全局变量
- 渲染预览
- 发布范围
- 发布前检查

这两层分开以后，思路会清楚很多。

前者是在管“过去和现在配置长什么样”，后者是在管“你准备发出去的东西到底稳不稳”。

配置搜索和定时备份这两页，我觉得也别小看。前者是排障时很好用的全局检索入口，搜 IP、VLAN、路由、ACL、NTP、SNMP 会很顺；后者看着不显眼，其实是在回答“备份是不是持续发生、策略是不是还活着”这种特别实际的问题。

## Diff 这种能力，平时不显山露水，真出事会很值钱

很多时候工程师最想知道的不是“这台设备被改过”。

这句话没什么用。

更有用的是另一句：到底改了哪几行？

这两个问题听着接近，实际不是一个层级。

知道“被改过”，只是知道发生了动作。
知道“改了哪几行”，你才能真正开始判断这次改动和故障有没有关系。

NetPilot 现在这块已经不只是能把两个版本摆在一起看了。你可以先按设备或者快照去搜，再选左右两个版本做对比，还能按 interface、route、acl、bgp、ospf 这些关键词去快速定位差异块。

这就不太像“把 diff 命令搬进页面”了，而是真的在往排障和变更复盘的体验上靠。

## 合规审计这件事，会一下子把平台味拉出来

很多自动化项目做到“能批量执行命令”就停了。

可一旦往企业环境里走，大家关心的很快就不只是“能不能做”，而是“做得规不规范”。

这就是为什么合规页会特别像平台分水岭。

NetPilot 现在这块已经不是跑一遍检查、弹一个结果那么简单了。它在做的是把合规这件事变成一套长期对象：

- 有规则库
- 有检查结果
- 有发现项
- 有严重级别
- 有状态
- 有后续跟进

这里最重要的一点，是“发现项”不再是一闪而过的扫描结果。

审计跑完之后，不是你看个分数就结束了，而是还能继续按级别、状态、分类去筛，打开详情，处理后续动作。平台开始承接的，已经不是那一次扫描，而是扫描之后的整改链路。

## 企业环境很多时候怕的不是故障，是长期不规范

有些问题不会立刻把设备打挂，但会一直埋着。

比如：

- SNMP 没按标准开
- 日志和 NTP 配得不完整
- AAA 没落到位
- 管理平面有风险
- 某些交换特性留着隐患

这些东西人工不是看不了，是看不动，也坚持不住。

今天看一遍，明天忘了，后天又来了。

平台化审计最实在的地方，就是把“偶尔人工扫一下”变成“可以重复跑、可以持续跟、可以留下状态”的东西。

而且再往后看一步，合规和审计日志其实是连着的。合规页负责把问题揪出来，审计日志负责把谁做了什么动作记下来。两块一拼，平台才开始有点治理系统的意思。

## 拓扑这块也是，重点从来不是图好不好看

资产表当然重要。

但资产表本质上还是静态视角。你能看到设备名、IP、平台、站点，可很多时候现场最想知道的根本不是这些，而是关系。

比如：

- 这台设备上联是谁。
- 下联是谁。
- 它在网络里大概是什么位置。
- 哪两台设备之间现在到底还有没有邻接关系。

这种问题如果只靠表格去脑补，速度会很慢。

NetPilot 现在基于 LLDP 去做拓扑发现，本身不稀奇。让我觉得它开始像样的，是它已经不只是“把线连起来”。

现在这张图里开始带站点、角色、节点状态、链路健康、陈旧链路、多源证据，右边还能看当前节点的邻接、链路详情和优先关注对象，甚至还能往导出和后续分析走。到这一步，它才不只是展示图，而更像排障入口。

## 所以这几块决定的，其实是平台有没有后劲

如果把前两篇讲的内容理解成“能不能跑起来”，那这篇讲的更像“能不能稳住”。

配置管理解决的是：变更有没有基线。
Configuration 工作台解决的是：发出去之前有没有准备好。
合规审计解决的是：规则和整改有没有闭环。
审计日志解决的是：动作有没有留下来。
拓扑发现解决的是：关系和影响面能不能被看见。

这些东西一旦补上，平台就不会只停留在“我能连设备、我能发命令、我能看结果”这几个字上，而是开始有点正式环境需要的那种厚度。

一个网络运维平台真开始像平台，通常不是因为它终于会发命令了，而是因为它开始接得住配置、标准和关系这些更深的上下文。

下一篇就收口。

权限、审计、技术架构、落地场景，这几块放在一起看，基本就能判断 NetPilot 到底只是个好用工具，还是已经长出平台骨架了。

如果你自己也在做内部平台，可以顺手想一下：

- 你们现在最缺的是“能执行”，还是“执行之后有人兜底”？
- 平台里最容易被忽略的，究竟是页面功能，还是配置、标准、关系这些底层上下文？

这两个问题，也是第 4 篇想收口的地方。
